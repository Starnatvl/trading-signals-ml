# Контекст заказчика (из переписки 24.02–02.03.2026)

## 1. ТЗ — неизменно

- Задача: ML-модель генерирует прибыльные BUY/SELL сигналы из `rd_value` + OHLCV.
- ТЗ носит рекомендательный характер. Формальных изменений не было.
- "Главное — какой PnL в результате предсказаний, а как при этом именно работает модель — не важно."

## 2. Два датасета, два смысла `signal_barrier`

| Источник | Семантика `signal_barrier` | Подтверждено |
|----------|---------------------------|--------------|
| Clean Data (`processed_data.parquet`) | Triple-barrier target: `close[t+1]` vs 1% TP / 0.5% SL | Accuracy 98.1% vs формула ТЗ |
| dataset_rework (CSV) | `sign(rd_value.diff(1))` — RD-derived label | Accuracy 99.9% vs `sign(rd.diff)` |

- Заказчик (Арсений) о rework-разметке: "сигналы ошибочные в части направления, но относительно точные в местах расположения", "будто бы их надо инвертировать".
- Разметка генеративная — заказчик сам неудовлетворён результатами.

## 3. Целевая переменная — на наш выбор

- "Какой вариант разметки лучше — решать вам."
- "У вас вообще может быть абсолютно другое решение."
- Исходные значения `rd_value` есть, а как их обработать — решать нам.
- Можно использовать любые данные из ccxt.

## 4. Комиссия

- 0.1% на сделку (плавающая, но ориентир).

## 5. API-контракт

- Текущий legacy: N=1 (один вектор фичей), но можно зафиксировать N>1 (`lookback_steps`).
- Наше решение: warmup 60 баров истории, не торгуем пока нет стабильных 1-мин данных.
- Модели всё равно какая монета — обрабатывает только числовые данные.
- Каждая монета хранится отдельно (история не пересекается).
- В потоке одновременно десятки криптовалют.
- Данные поставляются по API в реалтайме: `rd_value` + OHLCV.

## 6. Deliverables (хакатон)

1. Готовая модель (файл с весами).
2. `inference.py` с функцией `predict(features)` -> 1 (BUY) / -1 (SELL) / 0 (HOLD).
3. API-сервер: `POST /predict`, `GET /health`.

## 7. Важные нюансы

- Не торговать на первых 10+ барах (warmup фичей).
- Данные с разрывами не подавать — модель ждёт стабильные 1-мин данные.
- Если по монете нет данных > 1 часа — чистить историю.
- `signal_ema` (EMA crossover) — фактический сигнал текущей системы, можно использовать для сравнения.
